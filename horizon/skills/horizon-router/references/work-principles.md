# 工作原则

> 本文档从 horizon-router SKILL.md 拆出，按需加载。执行任何工具链任务前应阅读本文件。

- 开始任务前，先排查用户给定目录下的可用资源（模型权重、数据集、配置文件等），优先使用本地已有资源
- 如果需要下载外部资源，必须先向用户确认，不要自行下载
- **精度相关任务必须使用真实数据**：凡涉及量化校准、精度调优、cosine similarity 分析、精度 debug 等意图，必须使用真实校准/评测数据，禁止用随机数据替代。仅性能评测场景不受此约束
- 修改量化、编译、导出、推理、预处理或评测配置后，必须做最小可运行验证
- 验证失败时说明命令、报错和原因，基于原因修改后重试
- **UCP 部署任务必须闭环验证**：生成或修复推理代码后，必须完成「编译 → 部署到板卡 → 运行推理 → 输出推理结果」的完整流程。仅生成代码或仅修复 bug 不算完成——必须在板端拿到实际推理输出（分类结果、检测框、延迟数据等）
- **编译前预检工具链**：涉及 C++ 编译的任务（如 UCP 推理代码），先验证交叉编译器（`aarch64-none-linux-gnu-gcc`）、CMake、SDK 头文件是否可用，再开始编写代码。不可写完代码后才发现无法编译
- **区分"模型结构分析"和"性能分析"**：
  - 用户要求"检查/识别 CPU 算子、分析模型算子分布" → 属于**模型结构分析**，应查看编译报告（HTML 报告中的算子分布）或模型计算图，列出哪些算子落在 CPU 上
  - 用户要求"分析推理延迟、测量吞吐量、定位性能瓶颈" → 属于**性能分析**，应使用 pftrace/perf 工具
  - 两者不可混淆——性能分析工具无法告诉你哪些算子在 CPU 上执行
