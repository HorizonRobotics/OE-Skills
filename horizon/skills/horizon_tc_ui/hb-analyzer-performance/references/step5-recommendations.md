# 第 5 步：提供建议

基于第 4 步的分析结果，给出针对性的优化建议。

## 优化方向

### 1. CPU 算子优化

**如果发现 CPU 算子**：

```
⚠️ 发现 CPU 算子：
- Resize (layer_12)
- Softmax (layer_45)

建议：
- 检查这些算子是否可以替换为 BPU 支持的版本
- 考虑调整模型结构避免这些算子
- CPU 算子会显著降低性能
```

### 2. 瓶颈层优化

**如果某层耗时过高**：

```
🔍 瓶颈层：layer_23 (Conv2D, 3.2ms, 占 21%)

建议：
- 检查该层的卷积核大小和通道数
- 考虑使用深度可分离卷积
- 评估是否可以减少通道数
```

### 3. 带宽优化

**如果 DDR 带宽过高**：

```
📊 DDR 带宽：450 MB/run (偏高)

建议：
- 减少中间层的特征图尺寸
- 使用更激进的下采样策略
- 考虑模型剪枝
```

### 4. TAE 空闲时间

**如果 TAE idle 时间高**：

```
⏱️ TAE 空闲时间：15% (调度效率可提升)

建议：
- 模型结构可能存在依赖瓶颈
- 考虑调整层的并行度
```

### 5. 硬件对齐优化（进阶）

当第 4 步的模型结构分析（§H）发现关键维度不满足 BPU 对齐要求时，提供以下建议。

**BPU Nash 系列对齐要求**：

| 维度 | 对齐要求 | 影响的算子 |
|------|---------|-----------|
| Channel | 16 对齐 | Conv2D, Linear |
| Head Dim | 64 对齐 | Multi-Head Attention |

**当 head_dim 不满足 64 对齐时**：

```
硬件对齐问题：head_dim={head_dim} 不满足 BPU 的 64 对齐要求

建议：
- 调整 head 数量使 head_dim = embed_dim / num_heads 满足 64 的倍数
- 示例：embed_dim=384, 当前 num_heads=8 -> head_dim=48 (不满足)
  方案 A：num_heads=6 -> head_dim=64 (满足)
  方案 B：调整 embed_dim=512 -> head_dim=64 (满足)
- 不满足对齐会导致 BPU 插入 padding，浪费计算资源
```

**当 channel 不满足 16 对齐时**：

```
硬件对齐问题：Conv2D 通道数 {channels} 不满足 16 对齐

建议：
- 将通道数调整到最近的 16 的倍数
- 例如：channels=100 -> 调整为 96 或 112
```

**当 attention 是主要瓶颈时（即使硬件对齐满足）**：

如果 §F Block 级分析发现 attention 模块占总延时比例较高（> 30%），即使 head_dim 满足 64 对齐，仍应建议减少 head 数量来降低计算开销。

```
Attention 计算开销优化：
- attention 模块占总延时 {attn_ratio}%，是最大瓶颈
- 当前配置：num_heads={num_heads}, head_dim={head_dim}
- 建议适当减少 head 数量以降低 attention 计算开销：
  减少 num_heads 可以线性降低 Q·K^T 和 softmax 的计算量
  示例：num_heads 从 {num_heads} 减到 {num_heads//2}，
  head_dim 从 {head_dim} 变为 {head_dim*2}（仍满足 64 对齐）
- 需配合精度验证确认对模型效果的影响
```

### 6. 量化精度优化建议（进阶）

当第 4 步的量化配置审计（§G + §I）发现高精度占比过高时，提供以下建议。

**重要区分**：

| 层面 | 关注点 | 对性能的影响 |
|------|--------|-------------|
| **内部计算精度**（§I，核心） | int16 vs int8 MACs 比例 | 直接决定 BPU 计算吞吐量 |
| **输入/输出精度**（§G，补充） | float32/float16 vs int8 比例 | 影响 DDR 带宽（数据搬运量） |

**注意**：fp16 是 J6P 等 Nash 系列平台的默认精度，不应被单独识别为"高精度配置问题"。真正值得关注的是 int16（比 int8 精度更高、计算效率更低）。

#### A. 内部计算精度优化（优先）

当 §I 分析发现 int16 相关 MACs 占比 > 30% 时，使用以下建议模板：

```
内部计算精度优化建议：
1. 模型 {int16_ratio}% 的 MACs 使用了 int16 feature 精度（仅 {int8_ratio}% 为 double int8）
2. int16 会降低 BPU 计算吞吐量，int8 是 BPU 最高效的计算精度
3. 审查量化配置文件（qconfig），检查哪些层/模块的 feature 被配置为 int16
4. 将非关键路径的 feature 从 int16 降级为 int8：
   - 中间卷积层和线性层尽量使用 int8
   - 仅在精度敏感的输出层（如最终分类/回归头）保留 int16
5. 重新量化后对比 int16 占比变化和模型精度影响
```

#### B. 输入/输出精度优化（补充）

当 §G 分析发现高精度输入占比高 **且** DDR 带宽偏高时，使用以下建议模板：

```
输入/输出精度优化建议：
1. 审查非关键输入（如辅助特征、mask、坐标变换矩阵等）是否可从 float32 切换为 int8
2. 检查是否有算子因 fallback 而保持高精度，评估是否可手动配置为量化模式
3. 精度与性能的权衡：关键输出保持高精度，中间计算路径尽量量化
4. 如果 SegmentLUT / FakeQuantize 操作数量多，可能意味着部分算子
   fallback 到高精度模式，应逐一检查其量化配置
```

## 通用建议

无论具体指标如何，总是提供：

1. **性能评价**：
   - "性能表现良好" / "存在优化空间" / "性能较差"

2. **对比参考**：
   - "相比同类模型，延时处于中等水平"

3. **下一步行动**：
   - "建议先优化 CPU 算子"
   - "可以尝试板端实测验证"
   - "性能已达标，可以部署"

## 注意事项

- 建议要具体，避免泛泛而谈
- 优先级明确（先解决 CPU 算子，再优化瓶颈层）
- 给出可操作的方向，而非抽象概念
